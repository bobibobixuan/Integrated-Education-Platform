# 综合教育平台 UI 规范

> 文档版本：v2.2.23  
> 更新日期：2026-06-04  
> 文档类型：界面与交互规范  
> 使用对象：前端、UI、测试、AI Agent

---

## 1. UI 目标

本项目 UI 的目标不是做炫技页面，而是：

- 让学生一眼看懂去哪学
- 让教师快速定位管理入口
- 在手机和桌面都保持可用
- 不用视觉层破坏业务规则

---

## 2. 角色边界

### 2.1 学生端 UI

学生端页面：

- 首页
- 登录页
- 个人中心
- 冒险地图
- 冒险作答
- 冒险结算
- 随机练习
- 极限挑战
- 排行榜
- 成就墙
- 学习记录
- PVP 大厅
- PVP 房间
- PVP 对战
- PVP 结算

设计原则：

- 产品化文案
- 情绪正向但不过度幼稚
- 高频入口直达
- 手机优先

### 2.2 教师端 UI

教师端当前为：

- `/app/admin`
- 单页面后台
- 内部 tab 切换

主要模块：

- 仪表盘
- 学生管理
- 排行榜
- 教学分析
- 错题统计
- 题库管理
- PVP 管理
- 导入中心

设计原则：

- 信息密度高，但结构必须清晰
- 优先可读性，不追求花哨效果
- 长表格和长日志优先局部滚动

---

## 3. 视觉系统

### 3.1 学生端主题

当前学生端主视觉基线：

- 浅色主题
- iOS 风格卡片化布局
- 高圆角
- 清晰层级

关键色：

- 页面背景：`#F2F2F7`
- 卡片背景：`#FFFFFF`
- 主色：`#007AFF`

字体：

```text
-apple-system, BlinkMacSystemFont, "SF Pro Display", "PingFang SC", "Microsoft YaHei", sans-serif
```

约束：

- 不要继续使用旧的 `var(--accent)` 风格写法去回退页面
- 不要引入 `linear-gradient`
- 不要引入 `backdrop-filter: blur()`
- 不要引入 `radial-gradient`

### 3.2 教师端主题

教师端当前通过 `[data-theme="admin"]` 在 `style.css` 中覆盖变量。

特点：

- 更偏中性管理后台风格
- 强调表格、卡片和信息摘要

## 4. 布局规范

### 4.1 学生端

要求：

- 首页、登录页、排行榜、竞技大厅、个人中心、冒险地图必须优先考虑手机布局
- 不允许只把桌面双栏强行缩窄
- 手机上优先单列、卡片分组、底部高频操作明确

### 4.2 教师端

要求：

- 后台页面不要叠三层头部
- 页面标题、说明、业务内容要合并收口
- 日志、成员表、排行榜、导入预览等长内容优先卡片内滚动

---

## 5. 页面级规范

### 5.1 首页

必须：

- 品牌标题明确
- 主入口优先级清晰
- 账号区和学习入口区分清楚

### 5.2 登录页

必须：

- 先解决登录动作清晰度
- 注册入口是否展示取决于后端注册开关

### 5.3 作答页

必须：

- 题干为主
- 选项区足够清晰
- 反馈不能遮挡主信息
- 下一步操作明确

### 5.4 结算页

必须：

- 关键结果一眼可读
- 分数、题数、连击、正确率层级明确

### 5.5 成就墙

必须：

- 展示完整成就列表，不能只剩空占位
- 明确区分已解锁与未解锁状态
- 至少展示总数、解锁数、解锁率和最近解锁成就
- 未解锁项要给出提示，不暴露内部条件字段名

### 5.6 PVP 房间页

必须：

- 房间摘要
- 成员状态
- 我的状态
- 排名
- 主操作按钮

### 5.7 PVP 对战页

必须：

- 当前题目始终占主区
- 倒计时显眼
- 排名与反馈在侧区或明确分组

### 5.8 教师端 PVP

必须：

- 房间列表可扫读
- 当前房间重点信息集中
- 日志和排名不无限拉长页面

---

## 6. 文案规范

学生端文案：

- 用自然产品名称
- 不暴露实现细节
- 不写“导航分层”“第一屏”“组件”这类设计者语言

教师端文案：

- 优先准确、直接
- 避免过度营销化表述

---

## 7. 交互规范

### 7.1 按钮

- 主按钮只能有一个最强焦点
- 危险按钮必须弱于主路径按钮，除非当前动作确实高风险

### 7.2 表单

- 创建和编辑表单优先保持字段顺序稳定
- 缺省值应可理解

### 7.3 状态反馈

- 加载态要有明确反馈
- 失败态要能指导下一步
- 成功态不要过度打断流程

---

## 8. 状态边界

前端 UI 允许：

- 本地 loading
- 过渡动画
- UI 层反馈
- 非权威临时排序和展示

前端 UI 不允许：

- 擅自改变服务端返回的权威结果
- 擅自伪造房间最终状态
- 用样式重构破坏原有流程

---

## 9. 组件与样式边界

修改页面时需要同时考虑：

- 页面 `.vue`
- `frontend/src/style.css`
- 相关复用组件

禁止做法：

- 只改单页面，忽略全局视觉系统
- 把共享样式回退成旧主题
- 引入重量级 UI 库来覆盖已有界面

---

## 10. 测试关注点

UI 改动后至少关注：

- 路由能否正常进入
- 页面是否有明显语法错误
- 关键按钮能否点击
- 手机端是否退回桌面式布局
- 长列表是否把页面无限撑长

最低验证：

- `npm --prefix frontend run build`

高风险改动建议再补：

- 组件测试
- Playwright 验证

---

## 11. AI Agent 修改 UI 的规则

AI 修改 UI 时必须：

- 不乱删页面结构
- 不一次性重构整站
- 不改无关页面
- 不破坏已有业务流程
- 先读 `docs/UI_SPEC.md`
- 如果页面依赖全局视觉系统，要一起检查 `style.css`
