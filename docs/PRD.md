# 综合教育平台 PRD

> 文档版本：v2.2.22  
> 更新日期：2026-06-04  
> 文档类型：产品需求基线  
> 使用对象：产品、开发、测试、UI、AI Agent

---

## 1. 产品定位

综合教育平台是一套面向局域网课堂、校内机房和兴趣社团场景的 Python 学习产品。

它的目标不是做一个通用 LMS，也不是做在线代码沙箱，而是把下面几类能力整合到一个适合课堂落地的系统里：

- 学习引导
- 闯关练习
- 即时反馈
- 课堂管理
- 多人对战
- 学习数据沉淀

---

## 2. 角色定义

### 2.1 学生端

学生端是主学习界面，负责：

- 登录和注册
- 浏览首页与学习入口
- 进入冒险闯关
- 进行随机练习与极限挑战
- 查看排行榜、成就与个人记录
- 参与 PVP 房间和实时对战

学生端不负责：

- 判题真值
- 战力最终结算规则
- 房间状态的权威判定
- 管理端配置项维护

### 2.2 教师端

教师端是课堂运营与内容管理界面，负责：

- 管理员登录
- 班级与学生管理
- 题库、单元、关卡维护
- 仪表盘与教学分析
- 错题统计
- PVP 房间控制与复盘
- 导入数据

教师端不负责：

- 直接修改数据库文件
- 绕过鉴权调用内部逻辑
- 以 UI 假设服务端未实现字段

---

## 3. 业务目标

当前阶段产品目标：

1. 为编程初学者提供低门槛、可持续推进的 Python 学习路径。
2. 让教师能快速搭建本地课堂，不依赖公网服务。
3. 用排行榜、结算页和 PVP 提升参与感。
4. 让教师能实时看到在线状态、进度、错题和竞赛表现。
5. 让关键结果以服务端记录为准，减少前端作弊和状态错乱。

非目标：

- 不做跨学校 SaaS 多租户平台
- 不做富文本课程 CMS
- 不做自由代码执行评测沙箱
- 不做高度可配置流程引擎

---

## 4. 学生端模块

### 4.1 登录与注册

目标：

- 让学生快速进入系统
- 支持教师控制是否允许自助注册
- 在共享设备场景下避免账号串号

要求：

- 支持登录
- 支持按开关决定是否允许注册
- 登录后优先恢复服务端进度
- 登出时清理本地学习缓存

### 4.2 首页

目标：

- 让学生在一屏内看懂“去哪学、去哪玩、去哪看记录”

要求：

- 提供明确的主入口
- 命名必须站在产品视角
- 不暴露实现术语、开发过程、提示词
- 手机端和桌面端都必须高频可用

### 4.3 冒险闯关

目标：

- 作为主学习模式，形成稳定进度推进

要求：

- 以单元和关卡组织内容
- 每次作答必须绑定有效 `play_session_id`
- 正误、顺序和会话状态由服务端校验
- 结算给出星级、分数、连击和反馈

### 4.4 随机练习

目标：

- 提供轻量刷题入口

要求：

- 可从首页快速进入
- 与主闯关进度并存，不替代主线
- 数据进入服务端统计

### 4.5 极限挑战

目标：

- 提供更强挑战性和重复游玩价值

要求：

- 与普通闯关逻辑区分
- 结果进入用户统计
- 与排行榜、个人记录联动

### 4.6 排行榜

目标：

- 提供班级内公开激励

要求：

- 提供战力榜和在线时长榜
- 未登录用户也可查看公开榜单
- 登录用户可看到自己的名次
- 不能因昵称重复造成串榜

### 4.7 成就与学习记录

目标：

- 让学生知道自己完成了什么、错在什么

要求：

- 展示成就墙
- 展示错题和历史记录
- 提供知识点反馈，便于复习

### 4.8 PVP 房间与对战

目标：

- 提供课堂竞赛、活跃气氛和阶段性复盘入口

要求：

- 学生可创建、加入、准备和离开房间
- 房间状态通过 WebSocket 同步
- 对战结算必须处理未答题项，避免挂机获利
- 题目切换不能被乱序消息覆盖

### 4.9 个人中心

目标：

- 集中展示账号与个人数据

要求：

- 展示基础信息和累计表现
- 支持修改昵称、密码等基础操作

## 5. 教师端模块

### 5.1 管理员登录

要求：

- 管理端未登录时默认不展示后台内容
- 登录后进入统一后台视图
- 首次默认管理员登录后应强制改密

### 5.2 仪表盘

要求：

- 展示学生数、题目数、在线人数、累计作答、平均正确率、平均战力等核心 KPI
- 展示趋势图和需要关注的学生/单元

### 5.3 学生管理

要求：

- 支持创建、编辑、禁用、恢复学生账号
- 支持批量导入
- 支持查看学生详情、进度、表现

### 5.4 题库管理

要求：

- 支持题目 CRUD
- 支持题目启停
- 支持按单元、关卡、类型筛选
- 支持知识点字段维护

### 5.5 单元与关卡管理

要求：

- 支持创建单元
- 支持创建关卡
- 明确流程：单元 → 关卡 → 题目

### 5.6 教学分析

要求：

- 提供关卡层统计
- 提供高错题统计
- 提供教师可行动信息，而不是只堆原始数据

### 5.7 PVP 管理

要求：

- 查看房间列表、成员、日志和比赛状态
- 创建和编辑房间
- 开始比赛、结束比赛、手动同步
- 查看结算复盘
- 导出已结束房间结果

约束：

- 不实现后端尚未提供的备注保存或高级开关字段

### 5.8 导入中心

要求：

- 支持 JSON 和 Excel 两条导入链路
- 导入说明清晰
- 失败时给出可定位反馈

## 6. 业务规则

### 6.1 判题

- 题目正误必须由服务端判定
- 前端不能把 `is_correct` 当作可信来源回写

### 6.2 会话

- 每次答题都必须绑定有效 `play_session_id`
- 服务端必须校验题目顺序、会话状态和提交幂等性

### 6.3 学习进度

- 登录后优先拉取服务端进度
- 本地缓存只能做补充，不能覆盖服务端真实状态

### 6.4 鉴权

- 学生端和教师端 token 可以在同一浏览器并存
- 各自只同步本端会话

### 6.5 PVP

- 房间关键状态变更前需要二次校验
- 房间复位时必须清理比赛相关时间字段
- 房间状态广播以完整快照为准
- 后到消息覆盖先到消息

---

## 7. 非功能要求

### 7.1 可用性

- 局域网课堂环境下应能稳定访问
- Windows 为主要维护环境
- 根路径必须正确重定向到 `/app/`

### 7.2 性能

- 常规课堂规模下应支持多人登录、答题、房间同步
- WebSocket 在线状态和 PVP 更新要保持可接受延迟

### 7.3 安全

- 不允许保留硬编码后门凭据
- 不允许跳过鉴权的调试入口
- 管理端内容不能不受控注入 `innerHTML`

### 7.4 可维护性

- 保持当前前后端分层
- 服务层不直接创建 `SessionLocal()`
- 服务层不直接 `commit/rollback`
- 关键改动后要做最接近范围的验证

### 7.5 兼容性

- 前端部署路径固定为 `/app/`
- 高优先级页面必须兼顾手机端

## 8. 模块边界

### 8.1 前端边界

前端负责：

- 页面展示
- 用户交互
- 本地过渡状态
- 请求与实时订阅

前端不负责：

- 权威判题
- 权威房间状态流转
- 最终结算逻辑
- 权限兜底

### 8.2 后端边界

后端负责：

- 鉴权
- 判题
- 统计
- 会话与房间状态
- 数据持久化
- WebSocket 广播

### 8.3 数据库边界

数据库负责：

- 保存用户、题库、记录、PVP、成就等持久状态

数据库不应被：

- 文档指导直接手改
- UI 功能绕过服务层直接操作

### 8.4 UI 边界

UI 可以重构布局、视觉和交互层级，但不能：

- 擅自改业务规则
- 擅自增加后端没有的字段承诺
- 用视觉改动破坏已有流程

### 8.5 测试边界

测试必须覆盖：

- 变更影响到的直接链路
- 关键业务规则
- 高风险状态切换

## 9. AI Agent 执行要求

AI 在本项目里必须遵守：

- 先读文档，再读代码
- 不乱删
- 不大改
- 不改无关文件
- 不破坏已有功能
- 小步修改，小步验证
- 修改范围涉及产品、架构、接口、数据库或 UI 规则时，同步更新对应文档

- 持续打磨教师后台多页面与业务 Store 的维护体验
- 完善对外文档和 AI 协作文档，降低接手成本
