# 版本发布策略

> 面向项目维护者。  
> 本文档说明版本号规则和发版决策标准。

---

## 1. 分支定位

| 分支 | 定位 | 说明 |
|------|------|------|
| `main` | 正式稳定分支 | 只在一批功能收敛、测试通过、用户认可后推进 |
| `preview/dev` | 日常开发分支 | 所有 AI 修改、UI 调整、bug 修复、接口变动默认先提交到这里 |
| `feature/*` | 功能分支 | 较大功能的临时分支，如 `feature/pvp-admin`，完成后合并回 `preview/dev` |
| `fix/*` | 修复分支 | Bug 修复临时分支，如 `fix/login-redirect`，完成后合并回 `preview/dev` |

---

## 2. 版本号规则

采用语义化版本号：`v主版本.次版本.修订号`

```
v2 . 3 . 1
 │   │   └── 修订号（patch）：紧急修复、小 bug 修复
 │   └────── 次版本号（minor）：新增功能、UI 优化、小批量稳定改动
 └────────── 主版本号（major）：重大结构调整、完整模块重构、用户明确认可的大版本
```

---

## 3. 什么时候升级什么

### patch（修订号）`v1.0.0 → v1.0.1`

适用场景：
- 修复线上 bug
- 紧急安全修复
- 小范围 UI 修正
- 不影响功能的配置调整

示例：`v2.3.0 → v2.3.1`（修复 ad-blocker class 误判）

### minor（次版本号）`v1.0.0 → v1.1.0`

适用场景：
- 新增功能模块
- 页面 UI 优化
- 新增接口
- 小批量稳定功能合并

示例：`v2.2.32 → v2.3.0`（首次公开发布）

### major（主版本号）`v1.0.0 → v2.0.0`

适用场景：
- 重大架构重构
- 完整模块重写
- 用户明确认可的大版本节点
- 不兼容的 API 变更

示例：`v1.x → v2.0.0`（从旧架构迁移到 FastAPI + Vue 3）

---

## 4. 发版决策流程

```
功能开发完成
    ↓
在 preview/dev 测试
    ↓
判断改动规模
    ├── 小修复 → patch
    ├── 新功能 / UI → minor
    └── 重大重构 → major
    ↓
用户确认发版
    ↓
合并到 main
    ↓
更新 CHANGELOG.md
    ↓
打对应版本号 tag
    ↓
推送到 private + public
```

---

## 5. 版本号 vs 分支对照

| 分支 | 版本号策略 | 日志 |
|------|-----------|------|
| `preview/dev` | 不强制形成版本号，保留 patch 级流水 | `更新日志/` 按日期记录 |
| `main` | 每次合并必须打正式版本 tag | `CHANGELOG.md` 记录正式说明 |

---

## 6. 禁止事项

- 禁止在 `preview/dev` 上打正式版本 tag
- 禁止跳过版本号直接推 `main`
- 禁止同一个版本号打两次
- 禁止把 `preview/dev` 的详细日志直接当作 `CHANGELOG.md`
